The MMT API provides a number of extension interfaces that permit injecting customizing the behavior of MMT algorithms.
All these interfaces inherit from info.kwarc.mmt.api.frontend.Extension and can be studied by browsing the know abstract subclasses in the API documentation.
The API also includes numerous concrete subclasses.

Loading Extensions

Users load extensions with the extension shell command.
Programmers load them using the appropriate method of the ExtensionManager, which maintains all loaded extensions.

Writing Extensions

To write an extension, simply subclass the desired extension interface in Java or Scala.
The subclass must have a single constructor that takes no arguments.
Extensions are persistent and have full access to the MMT controller.

Extension Interfaces

The extension interfaces include
  • frontend.plugin adds general purpose functionality. In particular, plugins can be used to bundle multiple extensions or perform other customizations.
  • parser.ParserExtension adds new keywords to MMT's native text syntax. The MMT parser will relegate processing of declarations to the appropriate extension based on the keyword.
  • libraries.Foundation adds stand-alone type checking for a particular language.
  • archives.BuildTarget adds targets to the build tool.
    Extension developers only have to implement the action on a single file, and MMT takes care of recursion and change management.
  • presentation.Presenter implements rendering methods for new output formats.
  • web.ServerPlugin adds functionality to the HTTP server. Each ServerPlugin provides a key and serves requests to the URL /:key.
    Extension developers only have to implement the function, and MMT handles the HTTP protocol.
  • frontend.ChangeListener adds general purpose change-listening functionality. The methods of these extensions are called when MMT content is added, updated, etc.
  • ontology.QueryExtension adds a new atomic function to the MMT query language.
